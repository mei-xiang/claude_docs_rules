# 代码审查规范

## 审查标准

- **代码风格**：命名规范、格式一致性、注释质量
- **逻辑正确性**：边界条件处理、异常情况覆盖、逻辑错误检查
- **性能优化**：避免不必要的循环、合理使用缓存、数据库查询优化等
- **安全性**：输入验证、SQL注入防护、敏感数据处理、权限管理等

## 审查流程

- 提交前先自我审查
- 对照核心原则检查：是否破坏现有功能？是否最小化变更？
- 确认验证通过：编译通过、测试通过

## 反馈机制

提供具体的反馈，包括：
- 代码质量问题：指出具体位置和问题
- 修改建议：给出可操作的改进方案
- 性能优化建议：说明优化点和预期效果

## 审查清单

提交前逐项检查：

### 代码风格
- [ ] 命名是否符合规范（类名、方法名、变量名）
- [ ] 格式是否一致（缩进、空格、换行）
- [ ] 注释是否完整且准确

### 逻辑正确性
- [ ] 边界条件是否处理（空值、零值、极限值）
- [ ] 异常情况是否覆盖
- [ ] 审核代码背后的业务意图与代码实现是否一致，捕捉逻辑层面的错误，而不仅仅是语法错误

### 性能
- [ ] 是否存在循环查询数据库或重复计算
- [ ] 是否存在需要引入缓存机制而未使用的情况

### 安全性
- [ ] 是否验证输入参数（如类型、长度、格式）
- [ ] 是否存在 SQL 注入风险（如未使用参数化查询）
- [ ] 是否正确处理敏感数据（如密码、个人身份信息）
- [ ] 是否存在权限管理问题（如未授权访问）

### 变更范围
- [ ] 每行变更可追溯到用户需求
- [ ] 未引入不必要的依赖或文件
- [ ] 未顺手"优化"无关代码
